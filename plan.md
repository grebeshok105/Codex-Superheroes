# БЛОК 1. Орда → showcase-ready

Цель: орда выглядит зрелищно и «живо», без читерского ИИ, и **не ломается** (мобы не теряются, волна всегда завершается).

## 1.1. Фикс «мобы теряются» (корень бага)

**Причина:** `mobsAlive` — это просто `int`, который уменьшается ТОЛЬКО в `onMobKilled` (вызов из `BaseHordeEntity.die`). Если моб исчез не через `die()` (выгрузка чанка, `/kill`, провал в пустоту, баг деспавна, выход за far-away у не-Base сущностей) — счётчик навсегда застревает > 0, волна не завершается, приходится рестартить и искать мобов руками.

**Фикс — переход со счётчика на реальное множество сущностей:**
- В `HordeInstance` вместо `int mobsAlive` хранить `Set<UUID> liveMobIds` (UUID заспавненных тварей этой волны).
- При спавне (`spawnMob`) добавлять `entity.getUUID()` в `liveMobIds`.
- Каждый тик инстанса — **реконсиляция** (раз в ~10–20 тиков, не каждый тик, ради перфа):
  - для каждого UUID из `liveMobIds` дёргать `level.getEntity(uuid)`;
  - если сущность `null` / `isRemoved()` / `!isAlive()` → убрать из набора;
  - `mobsAlive = liveMobIds.size()` (для боссбара).
- `onMobKilled` оставить как быстрый путь (мгновенно убирает из набора при смерти), но **источник правды — реконсиляция**.
- Доп. страховка: если мобов в наборе нет живых и прошло > N секунд без событий — волна считается зачищенной.

**Бонус-страховка от «провалившихся» мобов:** в `BaseHordeEntity` уже есть телепорт к цели при застревании; добавим лёгкий «leash» — если моб дальше `SPAWN_RING_MAX + X` от центра орды ИЛИ упал ниже `center.y - 25`, телепортировать ближе к центру (а не только к таргету). Это убирает «мобы убежали/провалились» без всякого умного ИИ.

## 1.2. Спавн не толпой, а с разных сторон (направленный спавн)

Сейчас `spawnMob` — один случайный угол на кольце 20–35 блоков. Заменяем на распределение по секторам относительно **направления взгляда активатора** (или фиксированного «фронта» орды, см. вопрос Q1):

| Доля | Сектор | Откуда |
|------|--------|--------|
| 40% | фронт | сектор ±35° вокруг «передней» оси |
| 25% | фланги | левый/правый сектор (по ~12.5% каждый), ±(55°…125°) |
| 20% | тыл | сектор ±35° сзади |
| 15% | **с задержкой** | спавнятся не сразу, а через 1–3 секунды после старта волны (подкрепление) |

Реализация: при `startNextWave` раскидать `MobEntry`-список по 4 «корзинам» согласно долям; первые три корзины спавнить сразу в своих секторах, четвёртую — складывать в очередь `List<DelayedSpawn>` в `HordeInstance` с таймером и спавнить в `tickInstance`. Радиус кольца оставить 20–35 (или чуть варьировать по сектору). Тип моба в корзине сохраняется — просто меняется угол/тайминг, не состав.

Эффект: визуально «обкладывают со всех сторон» + «накатывают волнами», а не вылезают одной кучей. Никакого ИИ — только геометрия спавна.

# БЛОК 3. «Подумай, Марк» — жёсткий захват-замок (grab-lock)

Твоё требование: это **не «сделать плавнее»**, а **жёсткий замок**. Перестраиваю логику `OmnimanThinkMarkAbility`.

## 3.1. Новая модель фаз

Сейчас: авто LIFT→HOVER→DASH. Надо:

**Фаза HOLD (захват-замок), пока держишь / до ПКМ:**
- Цель **теряет управление**: для моба — `setNoAi(true)` + стоп навигации; для игрока-цели — клиентский input-lock (см. Q6) или просто силовой якорь.
- Каждый тик цель **ставится в якорную точку** перед руками игрока (примерно `eyePos + look*1.15`, центрируя по высоте хитбокса) — как сейчас в `holdTarget`, но усиленно.
- Скорость цели = скорость игрока: цель не двигается своим ходом, а «приклеена» — `setDeltaMovement(ZERO)` + перенос в якорь каждый тик.
- Цель **не падает отдельно**: `setNoGravity(true)` на время захвата, `fallDistance=0`.
- Хитбокс/коллизия временно не мешает: `target.noPhysics = true` на время HOLD (проходит сквозь блоки, не застревает, не отталкивается) + игрок может двигаться/летать с целью в руках, сохраняя **своё** управление.
- Снять `noPhysics`/`noGravity`/`noAi` строго при выходе из способности (важно — иначе моб останется призраком).

**Фаза DASH (по ПКМ):**
- По нажатию ПКМ (использовать предмет/спец-байнд — Q7) запускается принудительный совместный полёт **в сторону движения/взгляда** на ~3 секунды (~60 тиков).
- Летим вместе, **проламывая всё на пути** (уже есть `RushTerrainBreaker.breakContact`).
- В конце — `ShockwaveUtil.detonate` + бросок цели (как сейчас в `finish`), форс-падение.
- Урон оставляем небольшим — это зрелище (как и сказано в комменте файла).

## 3.2. Ключевые технические моменты

- **Игрок сохраняет управление в HOLD** (сейчас в LIFT/HOVER движение игрока форсится). Значит в HOLD не насилуем `player.setDeltaMovement` — двигается сам, цель следует за якорем.
- **Цель-моб:** `setNoAi`, `setNoGravity`, `noPhysics`, обнуление `xxa/yya/zza` и `deltaMovement` каждый тик. Это и есть «жёсткий замок» вместо «летит рядом».
- **Цель-игрок (PvP):** нельзя залезть в чужой инпут сервер-сайдом «насухо». Варианты: (a) силовой телепорт каждый тик + `ClientboundSetEntityMotionPacket(0)` (примерно держит, но игрок может рыпаться); (b) лёгкий клиент-мод инпут-лока через S2C-флаг (как уже есть `ThinkMarkS2CPayload`). Предлагаю расширить существующий payload флагом «захвачен» и глушить ввод клиентски. — Q6.
- **Безопасность:** если цель умерла/выгрузилась/сменила мир — мгновенно снять все флаги и завершить (как сейчас в `serverTick`).
- **Стартовое поднятие:** оставить короткий «рывок вверх» при захвате или нет? Сейчас LIFT тащит на 50 блоков вверх. По твоему описанию замок — горизонтальный («держишь в точке, по ПКМ летим в сторону движения»). Предлагаю **убрать авто-подъём** и держать цель там, где схватил, пока ты сам летаешь/двигаешься. — Q8.

## 3.3. Открытые вопросы по способности
- **Q6:** цель — только мобы или игроки тоже? Для игроков нужен клиентский инпут-лок (расширить `ThinkMarkS2CPayload`). Если только мобы — проще (NoAi решает всё).
- **Q7:** чем триггерить DASH — ПКМ предметом героя, отдельным байндом способности, или повторным нажатием той же кнопки?
- **Q8:** убрать авто-подъём на 50 блоков (держать на месте захвата) — да/нет?
