# v3.0 — User Fixes Plan

Документ-план только под пользовательские правки от 2025-01 (заявка из чата). **Без рефакторинга и без правок, которых не просили.** Каждый пункт = root cause + предлагаемый фикс + scope/риск + затронутые файлы.

Легенда:
- **Scope**: S (≤1ч), M (1–3ч), L (3–6ч), XL (>6ч).
- **Риск**: low / med / high — вероятность регрессий или необходимости итераций.

## 1. Слендермен: не поворачивается + видно от 1-го лица

### Текущее поведение
- Анимации (idle/walk/sprint) уже есть и переключаются (см. `SlendermanCloakEntity.movementHandler`).
- Сервер каждый тик проставляет yaw на cloak-сущности (`SlendermanCloakController:84-102`):
  - `cloak.moveTo(..., yaw, 0f)`
  - `cloak.setYRot(yaw)`
  - `cloak.setBodyYaw(yaw)` (synced data accessor `BODY_YAW`)
- Клиент-сторона — `PlayerRendererSlendermanHideMixin` отменяет vanilla-render игрока в HEAD методе `PlayerRenderer.render(...)` если игрок — слендермен.
- Cloak-сущность всё ещё рендерится перед камерой в 1-м лице (отдельная entity), отсюда «видно модельку от первого лица».

### Root cause
**1а) Не поворачивается.** Корень в одном из двух (нужно подтвердить тестом):
- `GeoEntityRenderer` базово интерполирует поворот через `entity.yRotO → entity.getYRot()`. У нашего `SlendermanCloakEntity` `yRotO` явно НЕ обновляется отдельно, мы зовём `setYRot()`/`moveTo()` каждый тик. `moveTo` ставит и `yRotO=yRot` через `setOldPosAndRot()` — поэтому интерполяция всегда даёт «текущий» yaw, **без угла отставания**, что визуально читается как «застыл лицом в одну сторону, никогда не поворачивается». Плюс: GeckoLib для не-`LivingEntity` НЕ применяет body-yaw отдельно (`yBodyRot` есть только у `LivingEntity`), он просто читает `yRot`.
- Дополнительно: cloak `noPhysics=true`, без AI, не двигается своей моторикой → packet-tracker сервера может не отправлять delta-rotation если код считает позицию неизменной.

**1б) Видно от 1-го лица.** Cloak-сущность отдельная от player-а, mixin отменяет только player-render. В 1-м лице камера у игрока, cloak-сущность находится примерно в позиции игрока + рендерится спиной к камере (если бы поворот работал) — а с учётом «не поворачивается» — вообще куда попало. Нужно явно скрыть cloak-renderer, когда камера == владелец cloak-а И камера в FIRST_PERSON.

### Предлагаемый фикс
**Поворот** (комбо из двух шагов, дешевле и надёжнее по отдельности):
1. В `SlendermanCloakEntity` хранить **previous body yaw** (`prevBodyYaw`) и обновлять в `tick()` *перед* применением нового; затем переопределить `getViewYRot(float partialTicks)` чтобы возвращать интерполяцию `prevBodyYaw → BODY_YAW`. Это даёт корректное interpolation окно для рендера.
2. В `SlendermanCloakRenderer.preRender(...)` (override hook GeckoLib) — подмешать поворот через `poseStack.mulPose(Axis.YP.rotationDegrees(180f - interpYaw))` если базовый GeoEntityRenderer не делает это сам для не-LivingEntity. Альтернатива — поднять иерархию cloak-сущности до `LivingEntity` (дороже, см. риски).
3. На сервере: использовать `cloak.setYHeadRot(yaw)` дополнительно к `setYRot`, и явно вызывать `cloak.lerpTo(...)` либо отправлять полную teleport-packet раз в N тиков чтобы избежать cap-а на rotation-delta.

**1-е лицо**:
- В `SlendermanCloakRenderer.shouldRender(cloak, frustum, camX, camY, camZ)` (или `addRenderLayers`/`render`) — early return если:
  - `Minecraft.getInstance().options.getCameraType() == CameraType.FIRST_PERSON`
  - **И** `Minecraft.getInstance().getCameraEntity() == cloak.getOwner()` (владелец = тот же игрок, чьей камерой смотрим)
- Дополнительно скрыть и player-arms через мини-mixin на `ItemInHandRenderer.renderHandsWithItems` или `GameRenderer.renderItemInHand` если они показываются (правка только если действительно видны после фикса cloak-а).

### Scope / риск
- Поворот: **M, med** (нужно тестировать GeoEntityRenderer-поведение в multiplayer — поворот тела vs. поворот головы).
- 1-е лицо: **S, low**.

### Файлы
- `entity/SlendermanCloakEntity.java`
- `effect/SlendermanCloakController.java`
- `client/render/SlendermanCloakRenderer.java`
- `client/mixin/PlayerRendererSlendermanHideMixin.java` (возможно расширение на `ItemInHandRenderer`)

- В `SlenderTendrilsAbility:90` `SLENDERMAN_ATTACK` громкость `0.9`.
- В `SlenderPhaseStalkAbility:38, 62` — `SLENDERMAN_LIVING` `0.4`, `SLENDERMAN_BUSH` `0.4`.
- В `SlenderStaticFieldAbility:45-47` (ULT) — `SLENDERMAN_JUMPSCARE` `1.4` + `SLENDERMAN_ANGRY` `1.0`.

### Root cause
Громкости подобраны индивидуально и местами переборщили (особенно `STATIC` 1.4 при maxStacks).

### Предлагаемый фикс
Глобальная константа в `ModSounds.SLENDERMAN_VOLUME_SCALE = 0.5f` + helper `playSlender(level, x,y,z, sound, volume, pitch)` который умножает на scale. Заменить все вызовы `level.playSound(... ModSounds.SLENDERMAN_*, ...)` через helper.

Эффект: всё тихнет в 2 раза. Лёгкая итерация — поднять scale до 0.6/0.7 если пользователь скажет «слишком тихо». Один мердж-конфликтный файл (`ModSounds`), точечные правки в 4-5 классах.

Альтернатива — поправить значения по месту, но тогда любая будущая правка громкости снова разбросана. Я бы делал helper.

### Scope / риск
**S, low** (15-30 минут).

## 4. Урон лазерам Хоумлендера ×10

### Текущее поведение
- `EyeLasersAbility`: `MIN_DPS = 14, MAX_DPS = 30` (per-second). Пер-тик урон = `dps / 20`. Madness множитель `MADNESS_DAMAGE_MUL = 3`. Итого без madness: 0.7–1.5 dmg/tick → ~14–30 dps.

### Root cause
По задумке масштабировался под мирных мобов — слишком слабо для бойцов с armor/боссов.

### Предлагаемый фикс
Поднять константы:
- `MIN_DPS = 140` (было 14) → 7 dmg/tick
- `MAX_DPS = 300` (было 30) → 15 dmg/tick

Также **обязательно** проверить:
- Hold-флаг урон через `target.invulnerableTime` — при таком dps (15/tick) `target.hurt()` вызовы каждый tick, а у LivingEntity `invulnerableTime=10` после удара. Нынешний код стреляет каждый tick — большая часть урона проглатывается i-frames-ами. **Пред-фикс**: либо обнулять `target.invulnerableTime = 0` после успешного `hurt()`, либо стрелять раз в `LivingEntity.invulnerableDuration / 2` тиков с накопленным amount-ом.

Без анти-i-frame фикса ×10 урон на бумаге даст ~×3 в реальности.

### Scope / риск
- Только константы: **S, low**.
- Константы + i-frame fix: **S, low-med** (риск — уронить других мобов через анти-iframe слишком сильно).

### Файлы
- `ability/EyeLasersAbility.java`

**Per-player flight speed multiplier** (5%-step, 50%–150%, дефолт 100%).

**Сторона сервера**:

Применять только горизонтальные компоненты (или с y тоже — см. Q3 ниже).
- Новый payload `FlightSpeedC2SPayload(int delta)` — клиент шлёт изменение (+5 / -5).
- Сохранять persistent (через AttachmentRegistry persistent + copyOnDeath).

**Сторона клиента**:

**Persistence**: значение сохраняется при де-трансформации/смерти/реконнекте (см. Q1).

Самый верный путь — отдельный server-tick хук: после `player.tick()` помножить `getDeltaMovement()`.

## 7. Помехи (static interference) застревают, фикс — только релог игры

### Текущее поведение
- `SlendermanStaticController` ведёт `LOOKER_STACKS: Map<UUID, Map<UUID, Integer>>` на сервере.
- Клиент держит `ClientSlenderState.staticStacks/staticFade` — **статические singleton-поля** на JVM.
- Сервер шлёт `SlenderStaticS2CPayload(stacks, fade)` каждые 5 тиков пока `processLooker` запущен (`SlendermanStaticController:115-117`).
- `processLooker` запускается только если `looker.tickCount % 5 == 0 || % 20 == 0` И только когда есть Slendermen. Если Slendermen нет — выход на `if (slendermen.isEmpty()) { LOOKER_STACKS.clear(); ... }` (line 61) — но **payload-ы клиенту в этом случае НЕ отправляются**.

### Root cause
Два пути к застреванию:
1. **Сервер**: Слендермен умер / де-трансформировался / вышел из мира → server-side state очищается, но **никакого финального `Payload(0,0)` не уходит** клиентам, у которых stacks > 0. Client-side `ClientSlenderState.staticStacks` остаётся равным «последнее значение» (например 17 → close-static текстура показывается forever).
2. **Клиент**: `ClientSlenderState` — JVM-singleton, не очищается на disconnect (выход в главное меню) и не очищается на смену мира. Только `Minecraft.stop()` (полный выход из игры) сбрасывает state. Это в точности соответствует репро пользователя — «фикс только перезаходом в игру, не в мир».

То же поведение для `fieldInside / fieldExpiryTick` — если caster ушёл в момент когда жертва внутри радиуса, "leave" пакет не уйдёт.

### Предлагаемый фикс
**Сервер**:
- В `SlendermanStaticController.init()` дополнительно: на каждом тике найти всех `looker`-ов, у кого только что обнулились stacks (или у которых был ранее не-ноль, а сейчас slendermen-нет), и отправить `SlenderStaticS2CPayload(0, 0f)` единожды.
- Добавить tracking `LAST_SENT_STACKS: Map<UUID, Integer>` чтобы не спамить `(0,0)` каждый тик; шлём только при transition `>0 → 0`.
- В `clear(UUID id)` (которая зовётся при де-трансформации) — отправлять и `(0,0)` payload явно если игрок ещё онлайн.
- Аналогично починить `SlendermanFieldController` — гарантировать что `SlenderFieldS2CPayload(false, 0)` доходит до `last`-инсайдеров при `expire()`.

**Клиент**:
- `ClientPlayConnectionEvents.DISCONNECT` (Fabric API) → вызвать `ClientSlenderState.reset()` который обнуляет все поля.
- `ClientPlayConnectionEvents.JOIN` (или ResourceManager reload) — на всякий случай тоже reset.
- Метод `ClientSlenderState.reset()` (новый): `staticStacks=0; staticFade=0; fieldInside=false; fieldExpiryTick=0;` (jumpscare-поля удалены см. п.6).

### Scope / риск
**S-M, low** — простой fix, но требует тестирования на multiplayer (особенно server-side `(0,0)` payload).

### Файлы
- `effect/SlendermanStaticController.java`
- `effect/SlendermanFieldController.java`
- `client/ClientSlenderState.java`
- `client/SuperheroesClient.java` (зарегистрировать `ClientPlayConnectionEvents.DISCONNECT`)

## 8. Регулус Magnet → Freeze: накапливается не весь урон

### Текущее поведение
`RegulusGreedController.startMagnet` → `tickMagnet` → `releaseAndFreeze`:
- При `releaseAndFreeze` жертва замораживается на `FREEZE_TICKS = 200` (10s).
- На неё ставится `FreezeState`, в `init()` зарегистрирован `ServerLivingEntityEvents.ALLOW_DAMAGE` который **не пропускает** урон, а кладёт его в `queuedDamage` (line 95-101).
- Когда freeze истекает, `release()` (line 213) **проигрывает** список:
  ```java
  for (QueuedDamage q : queuedDamage) {
      v.hurt(q.source, q.amount);
      if (!v.isAlive()) break;
  }
  ```

### Root cause
**Invulnerability frames!** `LivingEntity.hurt(...)` после успешного применения ставит `invulnerableTime = 20` тиков. Дальнейшие `hurt(...)` вызовы внутри этой петли будут отброшены/уменьшены:
- vanilla `LivingEntity.hurt`: если `invulnerableTime > 10` (только-что-был-удар) — урон применится только если `amount > lastHurt`, и применится только разница `amount - lastHurt`.
- В нашем цикле подряд идут все queued-дамажи — большинство глотается i-frame-ами.

Дополнительно: `if (!v.isAlive()) break;` — нормальный guard, не баг.

### Предлагаемый фикс
**Вариант A (минимальный)** — обнулять i-frames между вызовами:
```java
for (QueuedDamage q : queuedDamage) {
    v.invulnerableTime = 0;
    v.hurt(q.source, q.amount);
    if (!v.isAlive()) break;
}
```
Это применит каждый сохранённый удар целиком.

**Вариант B (правильный по балансу)** — агрегировать урон по `DamageSource` + применить как ОДИН удар:
```java
Map<DamageSource, Float> aggregated = new HashMap<>();
for (QueuedDamage q : queuedDamage) {
    aggregated.merge(q.source, q.amount, Float::sum);
}
DamageSource biggest = ...; float total = aggregated.values().stream().reduce(0f, Float::sum);
v.invulnerableTime = 0;
v.hurt(biggest, total);
```
Минусы: source теряет точность (если били разные игроки), kill-credits смещаются.

**Вариант C (компромиссный)** — агрегировать **по источнику**, применять каждую сумму как один hit с обнулением i-frame между источниками. Сохраняет авторство киллов.

Предлагаю **C** по умолчанию, но если пользователь хочет «весь урон что был — пусть весь и приходит, без размышлений» — то **A**.

### Scope / риск
**S, low** (вариант A) / **M, low** (вариант C). Риск — балансовый: одно подряд-обрушение 100+ урона может моментально one-shot жертву; именно так и должна работать механика по концепту.

### Файлы
- `effect/RegulusGreedController.java`

## 9. Кэп: щит не возвращается + сломана модель

### Текущее поведение
`ShieldProjectileEntity`:
- Хранит `savedStack` и `savedHand`.
- При `discard()` → `remove()` (line 219) — если `savedStack` непустой, кладёт обратно в `savedHand` или `inventory.add` или `drop`.
- При `MAX_LIFETIME_TICKS = 200` (10s) — гарантированно discard + return.
- При `bounces >= MAX_BOUNCES (3)` или нет следующей цели — `returning = true`, летит к owner-у. При сближении <1.5 — discard + return.
- При `onHitBlock` — `returning = true` + reverse motion ×0.3.

`ShieldProjectileRenderer` рендерит item-stack `VIBRANIUM_SHIELD` через `itemRenderer.renderStatic(..., FIXED, ...)` с поворотами `Y(rot)` + `X(rot * 0.7)` и scale `1.4`.

### Root cause
**9а) Не возвращается «иногда»** — несколько подозрений:
- Если owner сменил измерение / умер / вышел во время полёта проектила — `getOwner()` возвращает null или неактивного entity. В `tick()`:
  ```java
  LivingEntity owner = this.getOwner() instanceof LivingEntity le ? le : null;
  if (returning && owner != null) { ... }  // если owner==null — просто не возвращается
  ```
  При `lifeTicks > 200` всё равно discard. Но в `remove()`:
  ```java
  if (this.getOwner() instanceof ServerPlayer sp) { ... }
  ```
  — если owner == null, **shield пропадает в никуда**.
- Если owner далеко за 10s — projectile никогда не догонит → `lifeTicks > 200` → discard → owner получает item обратно (если жив). Но пользователь визуально не видит «возврата», т.к. он просто появляется в инвентаре без анимации. Это может быть «невозврат» в восприятии пользователя, не баг.
- Если щит застрял в block (хотя `noPhysics`-флага нет, но `Projectile` базово вызывает `onHitBlock` → reverse). Может быть случай когда reverse-векторы зацикливают щит у стены.

**9б) Модель сломана** — рендер крутит item с `ItemDisplayContext.FIXED` + `pose.scale(1.4)` + `Axis.YP rot` + `Axis.XP (rot*0.7)`. Vibranium shield model — кастом многослойный (1474 строки JSON, multi-tintindex). Возможные проблемы:
- `tintindex` 0..5 без зарегистрированного `ItemColor` provider → все слои белые → разные слои сливаются в одно белое пятно.
- Многоэлементная модель с rotation вокруг двух осей одновременно (Y+X) → элементы пересекаются и flicker-ят.
- `FIXED` контекст — используется для item-frame, может игнорировать display-overrides модели.

### Предлагаемый фикс
**9а) Возврат**:
- В `tick()` если `lifeTicks > 80` (4s) и `returning=false` — **принудительно** установить `returning = true`. Не ждать всех bounces.
- В `tick()` если `owner == null` после `getOwner()` — попытаться `level.getPlayerByUUID(savedOwnerUuid)` (нужно сохранить UUID отдельно, т.к. `Projectile.getOwnerUuid()` есть). Если найден — `setOwner(player)`.
- В `remove()` — если owner null И savedStack non-empty → попытаться найти любого онлайн-игрока с тем же UUID и вернуть; иначе спавнить ItemEntity на месте (чтобы не пропадал щит).
- Уменьшить `MAX_LIFETIME_TICKS = 100` (5s) — чтобы пользователь быстрее видел возврат.

**9б) Модель**:
- Проверить наличие `ItemColor` registration для `VIBRANIUM_SHIELD` (если планировалось multi-tint). Если нет — заменить multi-layer-модель на простую (single-layer) **только для projectile-render**: создать отдельный model `vibranium_shield_projectile.json` с одной текстурой и юзать его в renderer-е через явный resource.
- Альтернатива — рендерить custom block-item-style модель с одним слоем (типа vanilla shield). Простой fallback, надёжный.
- Убрать дабл-rotation: оставить **только Y-axis spin** (как Captain America's shield в фильмах). Cleaner и без артефактов.

### Scope / риск
- 9а: **M, med** — возврат через UUID-lookup аккуратно, чтобы не задевать ItemEntity-spawn в edge cases.
- 9б: **M, med** — нужно тестово подтвердить что модель именно из-за tintindex/двух-осного вращения. Если фактическая причина другая — может потребоваться итерация.

### Файлы
- `entity/ShieldProjectileEntity.java`
- `client/render/ShieldProjectileRenderer.java`
- `assets/superheroes/models/item/vibranium_shield_projectile.json` (NEW, optional)

- Запустить `./gradlew runDatagen` (см. skill `datagen`) — сгенерирует `models/item/iron_man_reactor.json` ссылаясь на `superheroes:item/iron_man_reactor`.

**Вариант B (вручную)** — создать `models/item/iron_man_reactor.json`:

+ переименование текстуры. Без datagen.

## 11. Мои собственные идеи

### Идея A — **Slenderman Pages** (sub-механика для Слендера)

**Концепт**: пока герой = Слендермен, в радиусе 64 блоков **спавнятся 8 невидимых "page"-маркеров** на структурах/деревьях. Враги-mob-ы внутри радиуса периодически «видят» страницы и идут к ближайшей. Если mob достигает страницы — она исчезает, накладывая на mob `Mining Fatigue III` на 30s. Когда все 8 собраны — глобальный re-spawn pages cycle, при этом во время re-spawn-окна Слендермен получает буст: +50% damage и `SLENDERMAN_VICTORY` cue.

**Зачем**: даёт Слендермену пассивную «расставь карту, веди врагов туда» механику без прямого боя — соответствует канону.

**Scope**: M-L, med (новые AABB-сканеры, partycle-маркеры, controller).

### Идея B — **Hero Synergy Bonuses** (мульти-герой)

**Концепт**: когда два разных героя в одном измерении в радиусе 32 блоков друг от друга, они получают временный синергический buff (динамически, пока в радиусе):
- Naruto + Goku → ×1.2 ki/chakra regen (теплая дальневосточная связка).
- Iron Man + Captain America → +20% armor (Avengers).
- Homelander + Slenderman → fear-aura: моб-ы в радиусе 16 от любого из них теряют target и убегают (horror-pair).
- Регулус + Goku → counter-attack возвращает 50% урона по атакующему.

**Зачем**: даёт смысл играть в команде с разными героями, повышает re-playability в кооп-режиме.

**Scope**: M, med-low — один controller, конфигурируемая `Map<Pair<HeroId,HeroId>, Buff>`. Расширяется списком в будущем.
