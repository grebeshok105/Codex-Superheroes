## 2. Жёсткое разделение 5 летающих + 5 наземных

Сейчас в `SungJinwooController.summonInitialArmy` чередование через
`(i % 2 == 0)` — это 5/5, но **вперемешку по слотам**. Пользователь
хочет именно: 5 первых слотов полукруга — летающие, 5 следующих —
наземные. Визуально: ряд летает справа, ряд идёт слева; или
конкретный сценарий «верхний полукруг — flying, нижний — ground».

### Решение

В `summonInitialArmy` — заменить чередование на блочное:

```java
for (int i = 0; i < MAX_SHADOWS; i++) {
    boolean grounded = (i >= MAX_SHADOWS / 2);   // [0..4] flying, [5..9] ground
    spawnSlotShadow(level, player, i, MAX_SHADOWS, grounded);
}
```

При этом `computeSlotPosition` в `ShadowSoldierEntity` сейчас раскладывает
все 10 слотов в один полукруг сзади. Это ок: первые 5 (летающие) встанут
по одной половине дуги, наземные — по другой. Если визуально хочется
«верх / низ» (flying — выше дуга, ground — ниже дуга в той же половине),
тогда **дополнительно** в `computeSlotPosition` использовать только
слотов того же типа: для flying-слотов 0..4 разложить в полукруг
diapazon `[-π/2 .. +π/2]`, для grounded-слотов 5..9 — тот же diapazon.
Тогда обе линии встанут параллельно, одна выше другой.

Реализация в `computeSlotPosition`:

```java
int half = MAX_SHADOWS / 2;
int localIndex = isGrounded() ? (slotIndex - half) : slotIndex;
int localCount = half;
double slotAngle = (localCount > 1)
    ? (localIndex - (localCount - 1) / 2.0)
        * (Math.PI / Math.max(1, localCount - 1)) * 0.95
    : 0.0;
```

`MAX_SHADOWS` остаётся 10 (5 + 5).

## 3. Адаптивная скорость теней

### Цель

Тень должна догонять цель: если цель идёт пешком — быть чуть быстрее
неё; если цель летает быстро (фантом, аллай, другой Сон Джи Ву на
полёте Iron Man и т.п.) — становиться быстрее _этой_ цели.

### Реализация

В `ShadowSoldierEntity#aiStep` после выбора цели вычислять желаемую
скорость и применять `AttributeInstance` с временным модификатором.

```java
private static final ResourceLocation PURSUIT_MOD =
    ModId.of("shadow_pursuit_speed");

private void updatePursuitSpeed(@Nullable LivingEntity target) {
    AttributeInstance moveAttr = this.getAttribute(Attributes.MOVEMENT_SPEED);
    AttributeInstance flyAttr  = this.getAttribute(Attributes.FLYING_SPEED);
    moveAttr.removeModifier(PURSUIT_MOD);
    flyAttr.removeModifier(PURSUIT_MOD);
    if (target == null) return;

    double targetSpeed = target.getDeltaMovement().horizontalDistance();
    double targetMaxSpeed = target.getAttribute(Attributes.MOVEMENT_SPEED) != null
            ? target.getAttribute(Attributes.MOVEMENT_SPEED).getValue() : 0.25;
    double sample = Math.max(targetSpeed * 5.0, targetMaxSpeed); // эмпирика

    // База у тени: 0.30 movement / 0.40 flying. Минимум +20% к base.
    double mult = Math.max(1.20, sample / 0.30 * 1.15); // 15 % быстрее цели
    moveAttr.addTransientModifier(new AttributeModifier(
        PURSUIT_MOD, mult - 1.0, AttributeModifier.Operation.ADD_MULTIPLIED_BASE));
    flyAttr.addTransientModifier(new AttributeModifier(
        PURSUIT_MOD, mult - 1.0, AttributeModifier.Operation.ADD_MULTIPLIED_BASE));
}
```

Звать в `aiStep` _после_ `setTarget`. Кап скорости — поставить `min(mult, 6.0)`,
чтобы тень не мчалась со скоростью элитры на 100 блоков/сек по карте.
Также — пересчитывать раз в 10 тиков, не каждый тик, чтобы атрибут не
мигал.

### Дополнительно

- Базовые `MOVEMENT_SPEED` поднять с `0.30` → `0.35`, `FLYING_SPEED` с `0.40` → `0.55`.
  Уже это даст «всем быстрее» из задания.
- Y-component у летающих не зажимать: при погоне за фантомом
  важно лазать вверх-вниз. В `computeSlotPosition` это уже сделано
  (offset `+2.5` от owner), но при наличии цели — лезть вверх до уровня
  цели + 1 блок.

## 4. Урон не проходит — фикс damage-redirect

### Симптом

Пользователь жалуется: «урон не наносится никому, ни мне — ни теням,
просто бьёт и тычка проходит вникуда».

### Корневая причина

В `SungJinwooController#init`:

```java
DamageSource diverted = src != null
        ? player.damageSources().mobAttack(victim)   // ← victim это shadow
        : player.damageSources().generic();
victim.hurt(diverted, amount);
```

Источник damage создан с `victim` (тенью) как атакующим. Затем в
`ShadowSoldierEntity#hurt`:

```java
if (ent instanceof ShadowSoldierEntity other && hasSameOwner(other)) {
    return false;
}
```

— блокирует урон от теней с тем же owner. То есть **редирект сам себя
блокирует**: тень видит «меня бьёт другая тень того же хозяина» и
игнорирует.

### Решение

Передавать в `victim.hurt` исходный `source` как есть, без mobAttack(victim):

```java
victim.hurt(source, amount);
```

Если `source.getEntity()` — это игрок-владелец, дополнительно
обернуть его, чтобы тень не отбрасывала за «owner = атакующий»:

```java
DamageSource diverted = source;
Entity src = source.getEntity();
if (src != null && src.getUUID().equals(player.getUUID())) {
    // Урон от self-инфликта (например, утопление с лагом, fall on shadow):
    // используем generic-источник, чтобы тень не отказала по owner-фильтру.
    diverted = player.damageSources().generic();
}
victim.hurt(diverted, amount);
```

Также проверить, что **сам Сон Джи Ву** не имеет лишних `i-frame`-ов
сразу после редиректа: после `return false` ванильный hurt() не
будет считать его урон, но `invulnerableTime` всё равно устанавливается
в 20 тиков? — нет, `setLastHurtByMob` и `invulnerableTime`
устанавливаются _после_ ALLOW_DAMAGE → если возвращаем false, vanilla
`hurt()` не выставляет invulnerableTime. Должно быть ок.

### Тесты

После фикса:
- бить зомби тень → у тени HP падает; через 5–6 ударов умирает.
- бить Сон Джи Ву зомби → урон уходит на одну рандомную тень.
- если живых теней нет — урон проходит на самого Сон Джи Ву.

## 5. Mobs aggro на тени после удара

### Симптом

«ЕСЛИ МОБА УДАРИЛИ — ОН ПЕРЕАГРИВАЕТСЯ НА НИХ».

### Корневая причина

Скорее всего вытекает из пункта 4 — если урон по мобам не проходил
(damage source со shadow-as-attacker мог фильтроваться клиентом или
пробрасываться неправильно), то `lastHurtByMob` у моба не
устанавливается, и его `HurtByTargetGoal` не срабатывает.

### Дополнительная страховка

Если после фикса 4 проблема остаётся — в `MeleeAttackGoal` тени всё
работает по умолчанию. Нужно убедиться, что `doHurtTarget` действительно
дамажит:

```java
@Override
public boolean doHurtTarget(Entity target) {
    boolean ok = super.doHurtTarget(target);
    // Дополнительно явно прокинуть «меня атаковал этот моб»
    if (ok && target instanceof Mob mob) {
        mob.setLastHurtByMob(this);
        mob.setTarget(this); // силовой агр
    }
    return ok;
}
```

`setTarget(this)` в обход goal-системы — это **последняя линия**, если
`HurtByTargetGoal` по какой-то причине не подцепляется (например, у
моба нет такой goal-системы, или goal-priority перебит).

⚠️ ВАЖНО: после `setTarget(this)` на каждый удар голем перестанет
сразу же агрить на других врагов — это нормально для дизайна задачи.

### Тесты

- ударить мирного железного голема тенью → голем разворачивается на тень.
- зомби, видящий и игрока и тень: ударить его тенью → переключается на тень.

## 6. После Sacrifice — тени дублируются (баг)

### Симптом

«после взрыва теней все тени почему то продублировались».

### Корневая причина

`SacrificeAbility#performActivate` делает:

```java
List<ShadowSoldierEntity> shadows = SungJinwooController.aliveShadows(player);
// ... explosion logic ...
SungJinwooController.disbandAll(player);
```

А `disbandAll` **сбрасывает** флаг `SUMMONED`:

```java
public static void disbandAll(ServerPlayer player) {
    List<UUID> ids = ARMY.remove(player.getUUID());
    SUMMONED.remove(player.getUUID());   // ← вот тут
    ...
}
```

На следующем server-tick `tickPlayer` видит `!SUMMONED.contains` и
вызывает `summonInitialArmy(player)` → 10 свежих теней. Из-за чего
после ультимейта Sacrifice армия мгновенно полностью восстанавливается —
а пользователь видит «дублирование», потому что свежие тени появляются
поверх старых частиц / тел / эффектов.

### Решение

Разделить два сценария:

```java
/** Тени уничтожены сценарием (Sacrifice). НЕ авто-респавнить. */
public static void consumeArmy(ServerPlayer player) {
    List<UUID> ids = ARMY.remove(player.getUUID());
    // SUMMONED НЕ трогаем — это значит «эта сессия уже отстреляла»
    if (ids != null) discardAll(player, ids);
    broadcastArmyState(player, false, 0);
}

/** Полный сброс — игрок снял костюм / разлогинился. */
public static void disbandAll(ServerPlayer player) {
    List<UUID> ids = ARMY.remove(player.getUUID());
    SUMMONED.remove(player.getUUID());
    if (ids != null) discardAll(player, ids);
    broadcastArmyState(player, false, 0);
}
```

В `SacrificeAbility` — заменить вызов:

```java
- SungJinwooController.disbandAll(player);
+ SungJinwooController.consumeArmy(player);
```

Та же логика нужна для любого будущего ультимейта, который
«сжигает» теней. `disbandAll` оставить только для off-transform.

### Тесты

- Призвать армию (10 шт), кастануть Sacrifice → взрыв, теней 0,
  через 1 сек теней по-прежнему 0, новых не появилось.
- Снять плащ → армия деспавнится. Надеть снова → новая армия 10 шт.
